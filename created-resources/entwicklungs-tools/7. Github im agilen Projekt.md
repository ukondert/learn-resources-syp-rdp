### 7.1.3 Collaborators und Zugriffsverwaltung

In einem Schulprojekt ist klare Zugriffsverwaltung genauso wichtig wie eine gute Ordnerstruktur. GitHub erlaubt es, Projektbeteiligte als **Collaborators** mit unterschiedlichen Berechtigungen einzuladen und so den Zugang gezielt zu steuern.

#### Best Practices fГјr Collaborators

- **Least Privilege**: Vergib nur die Rechte, die der Nutzer wirklich braucht. Entwickler erhalten meist `Write`, Projektleitung oder Lehrer `Maintain` bzw. `Admin`.
- **Keine gemeinsamen Accounts**: Jeder arbeitet mit seinem eigenen GitHub-Konto. Das erhГ¶ht Transparenz und Nachvollziehbarkeit.
- **RegelmГӨГҹige ГңberprГјfung**: Entferne ZugГӨnge, wenn Teammitglieder das Projekt verlassen oder die Rolle sich ГӨndert.
- **2FA fГјr Organisationen**: Wenn das Projekt in einer Organisation liegt, sollte Zwei-Faktor-Authentifizierung (2FA) Pflicht sein.
- **Teams statt Einzelpersonen**: Bei GitHub-Organisationen ist es besser, Nutzer in Teams zu organisieren, statt sie einzeln als Collaborators hinzuzufГјgen.
- **Verwende Teams mit definierten Rollen**: `Owners`, `Maintainers`, `Contributors` helfen bei der Skalierung.

#### Rollen und Empfehlungen

| Rolle | Beschreibung | Empfehlung fГјr das Schulprojekt |
|-------|--------------|--------------------------------|
| **Read** | Nur Lesezugriff auf Code und Issues | FГјr Beobachter / Stakeholder, wenn benГ¶tigt |
| **Triage** | Issues und PRs verwalten, aber keinen Code pushen | FГјr Scrum Master / Projektleiter ohne Coding-Aufgaben |
| **Write** | Code pushen, Issues und PRs bearbeiten | FГјr die meisten Entwickler im Team |
| **Maintain** | Repository-Einstellungen ГӨndern, Branches verwalten | FГјr Teamleitung, Scrum Master oder Lehrer |
| **Admin** | Voller Zugriff, inkl. Sicherheits- und Berechtigungseinstellungen | Nur fГјr Projektverantwortliche und LehrkrГӨfte |

#### Collaborators einladen

1. Г–ffne dein Repository und gehe zu **Settings** вҶ’ **Collaborators and teams**
2. Klicke auf **Add people** oder **Add teams**
3. Suche nach dem GitHub-Benutzernamen oder Team
4. WГӨhle die passende Berechtigung aus (`Read`, `Triage`, `Write`, `Maintain`, `Admin`)
5. Schicke die Einladung ab

> <span style="font-size: 1.5em">:bulb:</span> **Merksatz:** Ein GitHub-Repository sollte mindestens einen Admin haben, aber nicht alle Teammitglieder brauchen Admin-Rechte. Der Standardfall im Schulprojekt ist: Entwickler = `Write`, Teamleitung = `Maintain`, Lehrkraft = `Admin`.

#### Zugriffsverwaltung im Schulprojekt

- **Vermeide zu viele Admins**: Admin-Zugriff sollte auf wenige verantwortliche Personen beschrГӨnkt werden.
- **Nutze Branch Protection zusammen mit Collaborators**: Dadurch kГ¶nnen Entwickler zwar pushen, aber nicht direkt auf `main` mergen.
- **Audit-Log prГјfen**: Bei Problemen zeigt das GitHub-Audit-Log, wer wann welche Einstellungen geГӨndert hat.

#### Alternative: Organisation + Teams

Wenn das Projekt in einer GitHub-Organisation liegt, ist das sicherer als einzelne Repository-Collaborators:

- Erstelle ein Team wie `schulbibliothek-devs`
- FГјge alle Entwickler diesem Team hinzu
- Vergib dem Team die Rolle `Write`
- Nutze ein separates Team `schulbibliothek-admins` fГјr `Maintain` oder `Admin`

> <span style="font-size: 1.5em">:warning:</span> **Achtung:** Jeder, der `Admin`-Rechte hat, kann Branch Protection Rules ГӨndern oder Collaborators verwalten. Diese Rolle sollte nur wenigen Personen vertrauenswГјrdig Гјbertragen werden.

### 7.1.4 Branch Protection Rules einrichten

Branch Protection Rules sind Sicherheitsmechanismen in GitHub, die verhindern, dass Code ohne QualitГӨtsprГјfung auf wichtige Branches (z.B. `main`) gelangt. Sie erzwingen Prozesse wie Code Reviews und automatische Tests.

> <span style="font-size: 1.5em">:warning:</span> **Achtung:** Ohne Branch Protection Rules sind alle CI-Status-Checks nur informativ вҖ” ein Merge ist auch bei roten Tests mГ¶glich. Erst mit aktivierten Rules wird die QualitГӨtssicherung wirklich erzwungen.

#### Konfiguration in GitHub

1. Navigiere zu deinem Repository вҶ’ **Settings** вҶ’ **Rules** вҶ’ **Rulesets**
2. Klicke auf **"New ruleset"** вҶ’ **"New branch ruleset"**
3. Vergib einen Namen, z.B. `Protect main`
4. Unter **"Target branches"** вҶ’ **"Add target"** вҶ’ **"Include by pattern"** вҶ’ `main` eingeben
5. Aktiviere die gewГјnschten Regeln (siehe Tabelle unten)
6. Klicke auf **"Create"**

#### Empfohlene Regeln fГјr die Schulbibliothek

| Regel | Beschreibung | Empfehlung Schulprojekt |
|-------|--------------|------------------------|
| **Require a pull request before merging** | Kein direkter Push auf `main` erlaubt | вң… Aktivieren |
| **Required approvals** | Anzahl der erforderlichen Review-Approvals | вң… Mindestens 1 |
| **Dismiss stale pull request approvals** | Approval verfГӨllt bei neuem Push auf den Branch | вң… Aktivieren |

### 7.4.3 GitHub-hosted Runner vs. Self-hosted Runner

Workflows kГ¶nnen auf zwei Typen von **Runnern** ausgefГјhrt werden: auf von GitHub verwalteten Cloud-VMs oder auf einem eigenen, lokal betriebenen Rechner.

**GitHub-hosted Runner** sind virtuelle Maschinen, die GitHub fГјr jede Workflow-AusfГјhrung frisch bereitstellt. Nach Abschluss des Jobs wird die VM automatisch gelГ¶scht вҖ” keine Konfiguration notwendig.

#### VerfГјgbare GitHub-hosted Runner

|-------------------|----------------|-----|-----|-------------------|
| `ubuntu-latest` | Ubuntu 24.04 | 4 Cores | 16 GB | вң… 2.000 Min./Monat (Free) |
| `ubuntu-22.04` | Ubuntu 22.04 | 4 Cores | 16 GB | вң… enthalten |
| `windows-latest` | Windows Server 2022 | 4 Cores | 16 GB | вң… 500 Min./Monat (Free) |

> <span style="font-size: 1.5em">:bulb:</span> **Merksatz:** FГјr **Г¶ffentliche** Repositories sind GitHub-hosted Runner kostenlos und unlimitiert. FГјr **private** Repositories gilt ein monatliches Freikontingent вҖ” fГјr Schulprojekte in der Regel ausreichend.

**Self-hosted Runner** sind eigene Rechner (Laptop, Desktop-PC, Schul-Server), die als Runner bei GitHub registriert werden. Sie eignen sich, wenn spezielle Hardware/Software benГ¶tigt wird, Kosten gespart werden sollen oder Zugriff auf lokale Ressourcen (z.B. lokale Datenbank) notwendig ist.

#### Vergleich

| Kriterium | GitHub-hosted Runner | Self-hosted Runner |
|-----------|---------------------|-------------------|
| **Einrichtungsaufwand** | Keiner вҖ” sofort nutzbar | Einmalige Installation nГ¶tig |

| **Kosten** | Kostenlos fГјr Public Repos | Nur Stromkosten / eigene Hardware |
| **Isolation** | Frische VM bei jedem Run | Persistente Umgebung (Vorsicht: Zustand!) |
| **FlexibilitГӨt** | Standardumgebung, erweiterbar | Volle Kontrolle Гјber Software/Hardware |
| **Sicherheit** | Sehr hoch (isolierte VM) | Eigene Verantwortung |
| **Geeignet fГјr** | Standard-CI-Pipelines | Spezielle Anforderungen, lokale Ressourcen |
