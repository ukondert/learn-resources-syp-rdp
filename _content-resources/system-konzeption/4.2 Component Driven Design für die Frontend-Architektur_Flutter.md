## 4.2 Component Driven Design für die Frontend-Architektur

### Überblick

**Component Driven Design (CDD)** ist ein systematischer Ansatz zur Entwicklung von Benutzeroberflächen, der auf **Wiederverwendbarkeit**, **Konsistenz** und **Isolierung** setzt. Während DDD die Backend-Logik strukturiert, zeigt CDD, wie wir die UI-Logik im Frontend organisieren.

Das Besondere an CDD: Der **Backend First Ansatz** bedeutet, dass die Domain-Struktur aus DDD (Bounded Contexts, Aggregate) direkt die Frontend-Architektur prägt.

> <span style="font-size: 1.5em">🎯</span> **Ziel:** Sie lernen, wie Sie ausgehend von der Backend-Architektur eine strukturierte, wartbare Frontend-Architektur aufbauen, die auf wiederverwendbaren Komponenten basiert.

---

### 4.2.1 Was ist Component Driven Design?

#### Die drei Kernprinzipien

1. **Isolation:** Jede Komponente ist eine in sich geschlossene Einheit mit klaren Grenzen
2. **Komposition:** Komplexe UI-Strukturen entstehen durch Kombination einfacher Komponenten  
3. **Wiederverwendbarkeit:** Einmal entwickelte Komponenten können an vielen Stellen verwendet werden

#### CDD vs. Page-based Entwicklung

| Aspekt | Page-based | Component Driven |
|--------|-----------|-----------------|
| **Organisationsprinzip** | Nach Seiten | Nach Komponenten |
| **Wiederverwendung** | Copy & Paste | Komponenten von Natur aus wiederverwendbar |
| **Konsistenz** | Schwer zu gewährleisten | Automatisch durch shared Components |
| **Skalierbarkeit** | Unübersichtlich bei Wachstum | Skaliert gut durch Modularität |

#### Beispiel

```
// Page-based (traditionell)
pages/
  home.html      → eigener Suchcode
  catalog.html   → duplizierter Suchcode (Copy & Paste!)
  admin.html     → nochmal Suchcode kopiert

// Component Driven
components/
  SearchField.tsx  → wiederverwendbare Komponente
pages/
  HomePage.tsx     → nutzt <SearchField />
  CatalogPage.tsx  → nutzt <SearchField />
  AdminPage.tsx    → nutzt <SearchField />
```

### 4.2.2 Atomic Design: Die fünf Ebenen der Komposition

Brad Frost's **Atomic Design** ist das Organisationsprinzip für CDD. Es definiert fünf hierarchische Ebenen:

```
Atoms (Grundbausteine)
  ↓
Molecules (einfache Gruppen)
  ↓
Organisms (komplexe Abschnitte)
  ↓
Templates (Seitenlayouts)
  ↓
Pages (finale Implementierungen)
```

#### Die Ebenen im Überblick

| Ebene | Beispiele | Charakteristik |
|-------|----------|-----------------|
| **Atoms** | Button, Input, Icon, Badge | Nicht weiter zerlegt; Grundelemente |
| **Molecules** | Suchfeld (Input+Button), Verfügbarkeitsbadge | Gruppen von Atoms; einfache Kombinationen |
| **Organisms** | Buchkarte, Navigationsleiste, Tabelle | Komplexe Abschnitte; kombinieren Molecules+Atoms |
| **Templates** | StandardPageTemplate mit Platzhaltern | Layout-Struktur ohne echte Daten |
| **Pages** | BuchkatalogPage mit echten Daten | Finale Seite; nutzt alle Ebenen |

#### Schulbibliothek-Übersicht

```
Atoms:
  CustomButton, InputField, BookCoverImage, StatusBadge

Molecules:
  SearchField, VerfuegbarkeitsAnzeige, AutorMitBild, LeihfristWarnung

Organisms:
  BuchKarte, AppNavigationBar, AusleihHistorieTabelle, BuchSuchformular

Templates:
  StandardPageTemplate (mit searchWidget, contentWidget, sidebarWidget)

Pages:
  BuchkatalogPage, AusleihVerwaltungPage, MeineAusleihenPage
```

#### Vorteile von Atomic Design

1. **Klare Hierarchie** – Jede Ebene hat definierte Verantwortlichkeiten
2. **Maximale Wiederverwendbarkeit** – Atoms in vielen Molecules, Molecules in vielen Organisms
3. **Konsistentes Design** – Zentrale Definition von Design Tokens
4. **Einfacheres Refactoring** – Änderungen an Atoms wirken sich automatisch aus
5. **Bessere Skalierbarkeit** – Neue Features durch Komponenten-Kombination

### 4.2.3 Smart vs. Dumb Components

Die Trennung zwischen **Smart Components** (Container Components) und **Dumb Components** (Presentation Components) ist fundamental für CDD.

#### Smart Components: Die Daten-Manager

| Charakteristik | Smart Component |
|---|---|
| **Aufgabe** | Daten laden, State verwalten, Geschäftslogik koordinieren |
| **API-Calls** | Ja |
| **State Management** | Ja |
| **Styling** | Minimal bis keins |
| **Widget-Typ** | StatefulWidget |
| **Beispiel** | `AusleihVerwaltungPage`, `BuchSucheContainer` |

**Verantwortlichkeiten:**
- Datenabruf über Repositories
- State Management (setState oder State-Management-Tools)
- Error Handling
- Orchestrierung von Child-Components
- Kein Styling!

#### Presentation Components: Die Anzeige-Spezialisten

| Charakteristik | Presentation Component |
|---|---|
| **Aufgabe** | Daten anzeigen, UI-Events emittieren |
| **API-Calls** | Nein |
| **State Management** | Nein (außer UI-State wie "ist Dialog offen") |
| **Styling** | Vollständig |
| **Widget-Typ** | StatelessWidget |
| **Beispiel** | `BuchKarte`, `AusleihListe`, `SearchField` |

> <span style="font-size: 1.5em">ℹ️</span> **Props erklärt:** Props (Properties/Constructor-Parameter) sind die **Schnittstelle** einer Komponente. Sie sind Daten und Callbacks, die der Parent an die Komponente weitergibt. Eine Presentation Component hat keine eigenen Datenquellen – alles kommt von außen über Props. Beispiel: `BuchKarte(buch: buchDTO, onAusleihen: () {...})` – sowohl `buch` als auch `onAusleihen` sind Props.

**Verantwortlichkeiten:**
- Empfangen alle Daten über Constructor-Parameter (Props)
- Emittieren Events über Callbacks
- Vollständiges Styling und Layout
- Hochgradig wiederverwendbar

#### Vorteile der Trennung

1. **Testbarkeit** – Presentation Components einfach zu testen
2. **Wiederverwendbarkeit** – Presentation Components überall nutzbar
3. **Wartbarkeit** – Klare Verantwortlichkeiten
4. **Unabhängigkeit** – Smart Component kann sich ändern, Presentation Components bleiben gleich

### 4.2.4 Component API Design

Jede Komponente definiert eine **API** – ihre Schnittstelle zur Außenwelt.

![Component API Design - Klassendiagramm](./images/component-api-design.svg "Component API Design zeigt Props, Events und Slots")


**Legende zum Diagramm:**
- **Props (Inputs):** Daten fließen vom Parent zur Komponente (z.B. `buch: BuchDTO`)
- **Events (Outputs):** Komponente meldet Aktionen an Parent (z.B. `onAusleihen()`)
- **Slots:** Parent kann Child-Widgets flexibel einbinden (z.B. `child`, `header`)

#### Drei Kommunikationskanäle

**1. Props (Inputs): Daten in die Komponente**

```dart
class BuchKarte extends StatelessWidget {
  // Required Props
  final BuchDTO buch;
  
  // Optional Props mit Defaults
  final bool zeigeAutor = true;
  final bool zeigeSignatur = true;
  
  // Optional Callbacks
  final VoidCallback? onAusleihen;
  final VoidCallback? onVormerken;
  
  const BuchKarte({...});
}
```

**Design-Prinzipien:**
- Typsicherheit – Nutze Dart's Type-System
- Required vs. Optional – Markiere essentielle Parameter als `required`
- Default-Werte – Nutze Parameter-Defaults

**2. Events (Outputs): Kommunikation nach außen**

```dart
final void Function(String buchId)? onAusleihen;
final void Function(String buchId, String action)? onAction;
final VoidCallback? onClose;
```

**Event-Naming:**
- `on...` Präfix verwenden
- Semantische Namen – Beschreibe **was** passiert ist, nicht **wie** reagiert wird
- Minimal Payload – Nur notwendige Daten

**3. Slots (Children): Flexible Content-Bereiche**

```dart
class CustomCard extends StatelessWidget {
  final Widget child;           // Einfacher Slot
  final Widget? header;         // Optional Header-Slot
  final Widget? footer;         // Optional Footer-Slot
  final List<Widget>? actions;  // Actions-Slot
}
```

**Vorteile:**
- Parent kann Inhalte flexibel einbinden
- Komponente bleibt agnostisch über Content

### 4.2.5 State Management in komponentenbasierten Architekturen

State ist jeder Datenwert, der sich zur Laufzeit ändern kann. Systematisches State Management ist essentiell.

#### Die drei State-Ebenen

```
1. Lokaler State
   ↓ (Wird zu breit verteilt)
2. Lifted State (im Parent)
   ↓ (Wird zu komplex/viele Components)
3. Globaler State (App-weit verfügbar)
```

**1. Lokaler State: Die einfachste Lösung**

**Wann verwenden:**
- Toggle-States ("Dialog geöffnet?")
- Input-Feld-Werte
- Lokale UI-Zustände

**Beispiel:**
```dart
class BuchDetailsModal extends StatefulWidget {
  // ...
  
  @override
  State<BuchDetailsModal> createState() => _BuchDetailsModalState();
}

class _BuchDetailsModalState extends State<BuchDetailsModal> {
  // ✅ Lokaler State: nur diese Komponente
  bool _zeigeVerlaengerungsInfo = false;
  bool _istFavorit = false;
  
  // UI nur dieser Komponente
}
```

**2. Lifted State: Gemeinsame Verwaltung**

**Wann verwenden:**
- Data, die mehrere Geschwister-Komponenten benötigen
- Synchronisation zwischen Componenten

**Beispiel:** Parent verwaltet ausgewähltes Buch, teilt mit BuchListe und DetailsPanel

**3. Globaler State: Für App-weite Daten**

**Wann verwenden:**
- Authentifizierter Benutzer
- App-Settings
- Warenkorb / Ausleihen
- Theme (Light/Dark Mode)

**Faustregel:**
- Mehr als 3 Ebenen Props-Passing → **Lifted/Global State nutzen**
- Props brauchen mehr als 2 Komponenten → **Global State**

#### Anti-Pattern: "Prop Drilling"

Wenn Props durch zu viele Ebenen weitergegeben werden, wird es ungewartet. **Lösung:** Global State verwenden statt Props ständig weitergeben.

### 4.2.6 Design System und Komponentenbibliothek

Ein **Design System** sichert UI-Konsistenz, Effizienz und Skalierbarkeit.

**Aufbau:** Design Tokens → Komponenten → Patterns

#### Design Tokens: Die Grundlagen

```dart
abstract class AppColors {
  static const Color primary = Color(0xFF2196F3);
  static const Color secondary = Color(0xFF4CAF50);
  static const Color error = Color(0xFFF44336);
}

abstract class AppSpacing {
  static const double xs = 4;
  static const double sm = 8;
  static const double md = 16;
  static const double lg = 24;
}

abstract class AppTypography {
  static const TextStyle headline1 = TextStyle(...);
  static const TextStyle bodyLarge = TextStyle(...);
}
```

**Vorteile:**
- Zentral definierte Werte – ändern wirkt sich überall aus
- Konsistenz garantiert
- Theme-Wechsel einfach (Light/Dark Mode)

#### Komponenten mit Design Tokens

Alle Komponenten nutzen die Design Tokens:

```dart
class PrimaryButton extends StatelessWidget {
  // ...
  @override
  Widget build(BuildContext context) {
    return ElevatedButton(
      style: ElevatedButton.styleFrom(
        backgroundColor: AppColors.primary,      // ✅ Token
        padding: EdgeInsets.symmetric(
          horizontal: AppSpacing.lg,             // ✅ Token
          vertical: AppSpacing.md,
        ),
      ),
      // ...
    );
  }
}
```

### 4.2.7 Von Backend-Aggregaten zu Frontend-Komponenten

**Backend First Ansatz:** Die Domain-Struktur aus Kapitel 4.1 gibt die Frontend-Architektur vor.

Nachdem wir die Grundprinzipien von CDD (Atomic Design, Smart/Dumb Components, API Design, State Management und Design System) kennengelernt haben, können wir nun verstehen, wie die Backend-Struktur die Frontend-Architektur prägt.

#### Mapping-Strategie: Drei Ebenen

**1. Strategische Ebene: Bounded Context → Feature Modul**

```
Backend: Ausleih-Kontext
  ↓
Frontend: AusleihModule mit 
  - presentation/pages/   (Smart Components)
  - presentation/organisms/  (Presentation Components)
  - presentation/molecules/
  - presentation/atoms/
  - data/repositories/
  - data/models/
```

**2. Taktische Ebene: Aggregat → Smart Component**

Ein **Aggregat** im Backend wird zur **Smart Component** (Container Component) im Frontend. Diese Komponente:
- Lädt die Daten des Aggregats vom Backend
- Verwaltet den lokalen State
- Orchestriert Child-Components
- Delegiert Geschäftslogik durch API-Calls

**3. Detail-Ebene: Entity/Value Object → Presentation Component**

**Entities** und **Value Objects** werden zu **Presentation Components**:
- Empfangen alle Daten über Props
- Emittieren Events über Callbacks
- Enthalten vollständiges Styling
- Sind hochgradig wiederverwendbar

#### API-Contracts: DTOs als Schnittstelle

DTOs sind reine Datenstrukturen ohne Geschäftslogik. Das Backend konvertiert Domain-Objekte zu DTOs, das Frontend nutzt DTOs als Props.

**Struktur:**
```
Ausleihe-Kontext:
  → AusleihDTO (Aggregat)
  → AusleihExemplarDTO (Entity)
  → Leihfrist: DateTime (Value Object)
```

> <span style="font-size: 1.5em">:warning:</span> **Kritisch:** Geschäftslogik im Backend duplizieren ist ein häufiger Fehler. Das Frontend darf nur UI-Logik enthalten.

### 4.2.8 Feature-Struktur: Schulbibliothek Beispiel

```
lib/
  design_system/
    design_tokens.dart           # Farben, Spacing, Typografie
    components/
      buttons.dart
      cards.dart
  
  core/
    services/
      auth_service.dart          # Globale Services
    providers/                   # State Management
  
  features/
    
    buchkatalog/
      data/
        repositories/
          buch_repository.dart
        models/
          buch_dto.dart
      presentation/
        pages/
          buchkatalog_page.dart           # Smart
        organisms/
          buch_liste.dart                 # Presentation
          buch_karte.dart                 # Presentation
    
    ausleihe/
      data/
        repositories/
          ausleihe_repository.dart
        models/
          ausleihe_dto.dart
      presentation/
        pages/
          ausleihe_verwaltung_page.dart   # Smart
        organisms/
          ausleihe_liste.dart             # Presentation
          ausleihe_karte.dart             # Presentation
```

**Wichtig:** Feature-Module sind **fachlich** geschnitten (nach Bounded Contexts), nicht technisch. Nicht nach "components/", "services/", "models/" als Top-Level strukturieren!

### 4.2.9 Testing in komponentenbasierten Architekturen

Im Component Driven Design ist **Testen** essentiell. Eine fehlerhafte `BuchKarte` beeinträchtigt Suche, Merkliste und Ausleihliste.

#### Testing-Strategie

1. **Unit Tests:** Einzelne Komponenten mit Mock-Props testen
2. **Integration Tests:** Mehrere Komponenten zusammen testen
3. **Behavioral Tests:** Nutzer-Verhalten prüfen (nicht Implementation Details)

**Kurze Erwähnung:** Flutter nutzt `flutter_test`, Mocking mit `mockito`, und `widgetbook` zur Komponenten-Dokumentation.

#### Best Practices

- Teste **Verhalten**, nicht **Implementation**
- Presentation Components sollten einfach zu testen sein (StatelessWidget!)
- Smart Components benötigen Mocking von Repositories
- Komponenten-Tests sind Dokumentation des erwarteten Verhaltens

### 4.2.10 Zusammenfassung: CDD als Bindeglied

**Component Driven Design verbindet:**

1. **DDD Backend** (Bounded Contexts, Aggregate, Entities)
   ↓
2. **CDD Frontend-Struktur** (Feature-Module, Smart/Dumb Components)
   ↓
3. **Atomic Design** (Atoms bis Pages für konsistente UI)
   ↓
4. **Design System** (Einheitliche Tokens und Komponenten)
   ↓
5. **State Management** (Lokaler → Lifted → Globaler State)

**Resultat:** Eine skalierbare, wartbare und konsistente Frontend-Architektur, die natürlich aus der Backend-Struktur wächst.

#### Häufige Fehler (zu vermeiden)

| Fehler | Lösung |
|--------|--------|
| Geschäftslogik im Frontend dupliziert | Nur im Backend, Frontend delegiert via API |
| Zu große Smart Components | Aufteilen in kleinere Smart + Presentation Components |
| Props werden zu tief weitergegeben (Prop Drilling) | Global State für übermäßig verteilte Daten |
| Design Tokens nicht zentral definiert | Alle visuellen Werte in Tokens-Datei |
| Keine klare Trennung Smart/Dumb | Streng die Regel anwenden: Nur Smart hat API-Calls |
| Testing wird vernachlässigt | Presentation Components sind einfach zu testen – nutzen Sie das! |

> <span style="font-size: 1.5em">:bulb:</span> **Merksatz:** Component Driven Design ist nicht kompliziert – es ist eine **Disziplin**. Klare Struktur, kleine wiederverwendbare Komponenten, strenge Rollen. Das zahlt sich aus, wenn das Projekt wächst.
